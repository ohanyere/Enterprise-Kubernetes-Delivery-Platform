# Deployment Validation

Deployment validation confirms that rendered Kubernetes manifests can become healthy workloads and receive traffic safely.

## Readiness Probes

Validate that workloads define readiness probes against an endpoint that reflects application readiness, not only process liveness. A pod should not receive traffic until dependencies, configuration, and startup checks are complete.

Expected result: newly started pods stay out of service endpoints until they can serve requests correctly.

## Rollout Readiness

Validate rollout status after a deployment, promotion, or rollback. Check desired replicas, updated replicas, available replicas, and any rollout conditions that indicate progress or failure.

Expected result: the workload reaches a complete rollout state before the change is considered operationally successful.

## Image Verification

Validate that the running pod image matches the overlay image tag or digest. Compare rendered manifest intent with the image reported by Kubernetes.

Expected result: the cluster is running the exact image Git says it should run.

## Replica Validation

Validate that each environment runs the expected replica count before autoscaling changes it. Confirm pod distribution and availability are adequate for the environment's reliability target.

Expected result: dev, staging, and prod have replica counts aligned with their operational purpose.

## Resource Validation

Validate CPU and memory requests and limits. Confirm requests are present so the scheduler, HPA, and capacity planning can make reliable decisions.

Expected result: workloads are schedulable, governable, and compatible with HPA behavior.

## Service Selector Validation

Validate that Service selectors match pod labels generated by the Deployment or Rollout. Confirm endpoints exist after pods become ready.

Expected result: traffic routes to the intended ready pods and does not silently point at an empty endpoint set.
