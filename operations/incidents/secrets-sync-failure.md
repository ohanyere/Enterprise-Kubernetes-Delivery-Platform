# Incident Runbook: Secrets Sync Failure

## Symptoms

- Application pods fail during startup because required environment variables are missing.
- Kubernetes Secret is absent or stale.
- ExternalSecret status reports sync errors.
- Pods show `CreateContainerConfigError`.
- AWS access errors appear in External Secrets Operator logs.

## Missing Kubernetes Secret

The workload consumes a Kubernetes Secret generated by External Secrets Operator. If that Secret is missing, pod startup can fail before the application runs.

Check:

- The expected Secret name referenced by the Deployment or Rollout.
- Whether the ExternalSecret exists in the same namespace.
- ExternalSecret status conditions.
- ArgoCD sync status for secret architecture manifests.

Recover by fixing the ExternalSecret definition or temporarily rolling back the workload reference if the secret contract changed unexpectedly.

## ExternalSecret Sync Issues

Sync failures usually indicate the operator cannot read the external source or cannot write the generated Secret.

Check:

- ExternalSecret `Ready` condition and recent events.
- SecretStore or ClusterSecretStore reference.
- Refresh interval and last sync time.
- Namespace and service account permissions.

Recover by restoring the ExternalSecret, SecretStore, or operator permissions through GitOps-managed configuration.

## AWS Access Failures

AWS access failures can come from missing IAM permissions, broken IRSA or pod identity configuration, or an incorrect Secrets Manager ARN.

Check:

- IAM role policy allows the expected `secretsmanager:GetSecretValue` access.
- Service account annotation or pod identity association.
- AWS region and secret name.
- External Secrets Operator logs for access denied or not found errors.

Recover by restoring least-privilege access to the intended secret only. Avoid broad wildcard permissions during incident response unless explicitly approved by the security owner.

## Application Startup Failures

When a generated Secret exists but the application still fails, the secret keys or expected environment variables may not match.

Check:

- Keys in the generated Kubernetes Secret.
- `envFrom.secretRef` or individual `secretKeyRef` wiring.
- Application logs for missing config messages.
- Recent changes to secret schema or application config expectations.

Recover by restoring the expected secret key contract or rolling back the application version that introduced the mismatch.
